## Previsão de Término + Geração Automática de Datas

### Visão Geral
Adicionar campo "Previsão de Término" na temporada e usar `startDate + expectedEndDate + gameFrequency` para pré-preencher automaticamente o gerador de jantares/anfitriões. Sem quebrar nada do fluxo existente.

### Como Funciona Hoje (revisão)
- O componente `AutoGenerateDialog` já existe e gera datas por frequência (semanal/quinzenal/mensal).
- Atualmente o admin digita manualmente: Data Início, Data Final, Frequência e participantes.
- A função `generateSchedule` distribui os participantes nas datas em round-robin (sorteio aleatório).
- O `endDate` da Season é usado apenas para registrar quando a temporada foi encerrada (não é previsão).

### Mudanças

**1. Modelo e tipos**
- `src/lib/db/models.ts` → adicionar `expectedEndDate?: Date` em `Season` (opcional, não quebra dados antigos).
- `src/types/season.ts` → adicionar `expectedEndDate?: string` em `SeasonFormValues`.

**2. Formulário da Temporada**
- `src/components/season/SeasonBasicInfo.tsx` → adicionar input `type="date"` "Previsão de Término" ao lado de "Data de Início" (opcional, sem validação de obrigatoriedade).
- `src/hooks/season/useSeasonFormInitializer.ts` → carregar/inicializar o campo.
- `src/hooks/season/useSeasonFormSubmitter.ts` → enviar `expectedEndDate` ao salvar (vazio = `undefined`).

**3. Banco de Dados (Supabase)**
- Migração: adicionar coluna `expected_end_date DATE NULL` na tabela `seasons`.
- Verificar `SeasonRepository.ts` para garantir mapeamento (camelCase ↔ snake_case).

**4. Geração Automática de Datas (núcleo do pedido)**
- `src/components/season/host-schedule/AutoGenerateDialog.tsx`:
  - Receber `activeSeason` (ou via prop) para ler `startDate`, `expectedEndDate` e `gameFrequency`.
  - **Pré-preencher** os campos do dialog com esses valores quando abrir.
  - Mostrar uma mensagem informativa: *"Datas e frequência foram preenchidas com base na temporada. Você pode ajustar se necessário."*
  - Manter os campos editáveis (admin pode sobrescrever pontualmente).
  - Se `expectedEndDate` não estiver preenchido na temporada, comportamento atual (campo vazio).

**5. Aviso Inteligente**
- Se o admin tentar abrir "Gerar Lista" sem `expectedEndDate` definido, exibir um toast sugerindo: *"Defina a Previsão de Término da temporada para gerar as datas automaticamente."* (não bloqueia, apenas avisa).

### O Que NÃO Muda (proteção contra quebras)
- O `endDate` real continua sendo gravado automaticamente ao encerrar a temporada (lógica em `useSeasonFinalization` intacta).
- O admin continua podendo encerrar a temporada antes da previsão.
- Temporadas existentes sem `expectedEndDate` funcionam normalmente.
- O sorteio aleatório dos anfitriões (round-robin com shuffle) permanece igual.
- Nenhuma alteração na lógica de jackpot, ranking, prêmios ou partidas.

### Fluxo Completo (resultado final)
1. Admin cria/edita temporada e preenche **Previsão de Término** (ex: 31/12/2026).
2. Admin vai em "Configuração de Anfitriões" → "Gerar Lista".
3. Dialog abre com **datas e frequência já preenchidas** automaticamente.
4. Admin seleciona os participantes e clica "Gerar Lista".
5. Sistema cria todas as datas de jantar/partida até o término previsto, sorteando anfitriões aleatoriamente.
6. Se a temporada for encerrada antes do prazo, o `endDate` real é gravado normalmente — a previsão fica apenas como referência histórica.

### Tratamento de Timezone
Conforme convenção do projeto, datas serão parseadas como locais (`new Date(string + 'T00:00:00')`) para evitar erros de fuso horário.
